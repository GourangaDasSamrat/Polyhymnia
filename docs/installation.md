# Installation Guide (Linux & macOS)

This guide covers what to install to build and run every service in
Polyhymnia: the Go gateway, the C++ randomizer, the Rust database
service, and the static frontend.

> This guide targets standard desktop **Linux** (Ubuntu/Debian-based
> distros, with notes for Fedora/Arch) and **macOS** (via Homebrew).
> package names and build constraints and is out of scope here.

## 1. What you need, at a glance

| Tool                              | Purpose                                    | Minimum version    |
| --------------------------------- | ------------------------------------------ | ------------------ |
| `protoc`                          | Compiles `quote.proto` into language stubs | 3.15+              |
| CMake                             | Builds the C++ service                     | 3.16+              |
| C++ compiler (g++/clang++)        | Builds the C++ service                     | C++17 support      |
| gRPC + Protobuf C++ dev libraries | Linked into `cpp-engine`                   | matches `protoc`   |
| Go                                | Builds/runs `go-gateway`                   | 1.22+              |
| Rust + Cargo                      | Builds/runs `rust-db`                      | stable toolchain   |
| `just`                            | Runs the recipes in the `justfile`         | any recent version |
| Python 3                          | Serves the static frontend (`http.server`) | 3.x                |

## 2. Linux (Ubuntu / Debian)

```bash
sudo apt update
sudo apt install -y build-essential cmake pkg-config \
    protobuf-compiler libprotobuf-dev \
    libgrpc++-dev protobuf-compiler-grpc \
    python3
```

> **Note:** on many Debian/Ubuntu releases, `libgrpc++-dev` ships
> `pkg-config` (`.pc`) files rather than CMake config packages. The
> project's `cpp-engine/CMakeLists.txt` already handles this — it tries
> `find_package(... CONFIG)` first and automatically falls back to
> `pkg-config` if the CMake config isn't found. You shouldn't need to do
> anything extra, but if `just build-cpp` still fails to locate `grpc++`,
> verify with:
>
> ```bash
> pkg-config --modversion grpc++
> ```
>
> If that command reports "not found," see
> [Troubleshooting](troubleshooting.md#grpc-not-found-on-linux).

### Go

```bash
curl -LO https://go.dev/dl/go1.22.5.linux-amd64.tar.gz
sudo tar -C /usr/local -xzf go1.22.5.linux-amd64.tar.gz
echo 'export PATH=$PATH:/usr/local/go/bin' >> ~/.bashrc
source ~/.bashrc
```

Check the latest release at <https://go.dev/dl> if a newer version exists.

### Rust

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
```

### just

```bash
cargo install just
```

### Fedora / RHEL-based

```bash
sudo dnf install -y gcc-c++ cmake pkg-config \
    protobuf-compiler protobuf-devel \
    grpc-devel grpc-plugins \
    python3
```

### Arch-based

```bash
sudo pacman -S base-devel cmake pkg-config protobuf grpc python
```

## 3. macOS (Homebrew)

Install [Homebrew](https://brew.sh) first if you don't have it, then:

```bash
brew install cmake grpc protobuf go rust just python3
```

`brew install grpc` pulls in Protobuf and the C++ gRPC libraries
together, and — unlike most Linux distro packages — Homebrew's gRPC
ships proper CMake config files, so `find_package(gRPC CONFIG REQUIRED)`
resolves directly without needing the `pkg-config` fallback.

If CMake still can't find gRPC after installing, make sure Homebrew's
prefix is on your `CMAKE_PREFIX_PATH`:

```bash
# Apple Silicon
export CMAKE_PREFIX_PATH="/opt/homebrew:$CMAKE_PREFIX_PATH"

# Intel Macs
export CMAKE_PREFIX_PATH="/usr/local:$CMAKE_PREFIX_PATH"
```

## 4. Go and Rust protoc plugins

The Go stubs are generated by two `protoc` plugins that aren't part of
`protoc` itself:

```bash
go install google.golang.org/protobuf/cmd/protoc-gen-go@latest
go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@latest
```

Make sure `$(go env GOPATH)/bin` is on your `PATH` (add to `~/.bashrc`
or `~/.zshrc`):

```bash
export PATH="$PATH:$(go env GOPATH)/bin"
```

Rust doesn't need a separate plugin — `rust-db/build.rs` calls
`tonic-build` automatically on `cargo build`, which shells out to
whatever `protoc` you installed above.

## 5. Verify your toolchain

```bash
protoc --version
cmake --version
g++ --version        # or clang++ --version
go version
cargo --version
just --version
python3 --version
```

All of these should print a version with no errors.

## 6. Build and run

From the repository root:

```bash
just build   # generates gRPC code and compiles rust-db, cpp-engine, go-gateway
just run     # builds everything, then launches all four services together
```

Then open **http://localhost:5500** in a browser and click **Get Quote**.

Individual services can also be built/run on their own — see the
[README](../README.md#project-layout) for the full list of `just`
recipes, or run `just --list`.

If something goes wrong along the way, check
[Troubleshooting](troubleshooting.md) before opening an issue.

## 7. Run with Docker (automatic)

If you'd prefer an automatic, self-contained way to run Polyhymnia, use
the provided Docker configuration. This builds each service into a
container and runs them together via `docker-compose`.

Prerequisites:

```bash
docker --version
docker-compose --version   # or `docker compose version` for newer Docker
```

Quick start (build and run):

```bash
docker-compose build
docker-compose up
```

Services will be available at the same ports as the local/manual setup:

- Frontend: http://localhost:5500
- Gateway: http://localhost:8080/api/quote
- Rust gRPC: localhost:50051
- C++ gRPC: localhost:50052

Stop and remove containers:

```bash
docker-compose down --rmi local
```

Notes:

- The `docker-compose.yml` builds images from the repository subfolders
  `rust-db`, `cpp-engine`, `go-gateway`, and `frontend`. If you want to
  use prebuilt images instead, replace the `build:` sections with
  `image:` references.

## 8. Manual (existing) — for development and debugging

The repository's existing `just` recipes remain available for
developers who prefer building and running services locally (the same
steps the Docker images implement). Use `just build` and `just run` as
documented above for the manual workflow.

Use Docker for an automated, reproducible deployment; use the manual
`just` workflow when iterating on code locally.
